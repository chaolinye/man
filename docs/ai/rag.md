## 1. 为什么需要 RAG？

先想一个问题。一个普通的 LLM——比如 GPT-4——你问它「2026 年的诺贝尔物理学奖得主是谁？」

如果它的训练数据只到 2025 年底，它可能：
- 胡编一个答案（这叫**幻觉**）
- 或者直接说「我不知道，我的知识截止于……」

LLM 本质上是把训练时看到的几十 TB 文本「压缩」成了一个巨大的神经网络。它的大脑里没有「书」，没有「数据库」，没有「能查资料的地方」——只有训练时形成的模式和关联。

**RAG 要解决的核心问题就是：怎么让 LLM 在回答问题的时候，能去查资料？**

就这么简单。跟你在考试时开卷一样。LLM 本来是一个闭卷考试选手——凭记忆作答。RAG 给它一本参考书。

## 2. 最基础的 RAG 流程

### 第一步：准备「参考书」（索引阶段）

假设你有一堆文档——公司的 Wiki、技术手册、或者你自己写的笔记。LLM 训练时没看过这些。你想要它回答关于这些文档的问题。

首先你得把这些文档变成它能「查」的东西。怎么做？

**切成小块。** 你不能把整本 1000 页的书扔给它——LLM 的「工作记忆」（上下文窗口）有限。所以你切。

> 想象你有一本百科全书，你在关键页贴了便签。你要回答问题的时候，不是把整本书从头读到尾——你去找相关的几页。

这就是 **chunking（分块）**。每块大概 256-1024 个 token，可以重叠一点避免在关键地方被切断。

**然后：把每一块变成数字。**

你不能直接用文字搜文字，效率太低了。你给每一段文字做个「数学签名」——一个数字列表，叫 **embedding（嵌入向量）**。这个数字列表的特点是：**意思相近的文字，数字列表也相近。**

```
「苹果好吃」和「苹果很甜」→ 两个数字列表之间的距离很小
「苹果好吃」和「熊喜欢吃蜂蜜」→ 距离很远
```

这些 embedding 是专门训练的模型生成的。

**把所有 embedding 存到一个数据库里。** 这个数据库叫 **vector database（向量数据库）**，它能做的事情就是：你给我一个向量，我快速找出跟你最接近的 N 个向量。

### 第二步：查书（检索阶段）

用户问了一个问题，比如：「苹果公司的第一款手机是什么时候发布的？」

你做三件事：

1. **把用户的问题也变成一个 embedding**——用的同一个模型，保证格式一致
2. **去向量数据库里找最相似的前 N 块**——比如找出最相关的 5 个 chunk
3. **把找出来的内容当作「参考资料」**

这就是检索阶段。本质上就是「找东西」。

### 第三步：回答问题（生成阶段）

现在你有了两样东西：
- 用户的原始问题
- 检索到的相关段落

你把它们拼在一起，组装成一个**新的提示词（prompt）**，然后把这个完整的 prompt 喂给 LLM。它看到参考资料了，从里面找到了答案。

**这就是最基本的 RAG。** 它就是把「查」和「写」拆成了两步，而不是让 LLM 全靠回忆瞎猜。

```
用户 query
    ↓ query embedding
向量数据库检索 → 返回相关 chunk 的原始文本
    ↓ query + chunks 拼成 prompt
LLM 生成回答
```

## 3. 分块（Chunking）—— RAG 的第一步

### 核心矛盾

文档是按「作者逻辑」写的，问题是按「用户逻辑」问的。分块就是为了在这两个逻辑之间搭桥。

**两个极端都不好：**

- **整篇文档作为一个块**：信息完整，但噪音太多，embedding 向量被稀释，检索精度下降
- **每句话作为一个块**：精确但信息不完整，大部分问题的答案需要多句话才能说清楚

分块是在两个极端之间找平衡：**信息完整度 vs 信息密度**。

### 方法一：固定字符数切分（别用）

```python
def fixed_size_chunk(text, chunk_size=500, overlap=50):
    chunks = []
    for i in range(0, len(text), chunk_size - overlap):
        chunks.append(text[i:i+chunk_size])
    return chunks
```

**问题：** 它可能在句子中间、单词中间甚至汉字中间切一刀。第一块末尾话说了一半，第二块开头前文丢了一半。

### 方法二：递归字符切分（Recursive Character Text Splitter）

这是 LangChain 默认的方式，也是大多数人的起点。它的思路是：**按优先级依次尝试不同级别的分隔符。**

```
优先级：
1. 段落分隔（\n\n）  → 尽量在段落之间切
2. 换行符（\n）       → 段落内部，在行之间切
3. 句号（。！？）      → 行内部，在句子之间切
4. 逗号分号           → 句子内部
5. 字符               → 最后手段
```

这是一种「保底」的方法——尽量在自然边界切，实在不行才强行切。

```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50,
    separators=["\n\n", "\n", "。", "！", "？", "；", "，", " ", ""]
)
```

### 方法三：语义切分（Semantic Chunking）

不按字符数切，按「这段话的主题变了没有」来切。

**做法：**

1. 把文档分成候选段（比如按句子或段落）
2. 把每个候选段变成向量
3. 计算相邻两段的语义相似度
4. 如果相似度突然下降 → 主题变了 → 切

这个方法的优点是切出来的每个 chunk 在语义上是连贯的。缺点是多了 embedding + 相似度计算的开销。

### 方法四：结构感知切分（Structure-Aware）

如果你知道你的文档是什么格式（Markdown、HTML、LaTeX），利用这个结构信息是最好的。

**做法：** 按照标题层级来切，每个 chunk 保留父标题作为元数据。

```python
# 伪代码
chunk = {
    "text": "修改 host 为 localhost...",
    "metadata": {
        "h1": "第三章：数据库配置",
        "h2": "3.2 连接配置",
        "page": 42,
    }
}
```

这样检索到 chunk 的时候，LLM 不仅看到内容，还看到「这个内容属于第三章 > 3.2 连接配置」。这个上下文信息对回答质量非常有用。

### 结构感知切分怎么控制 chunk_size？

这是三个约束的三角博弈：

```
按结构切（语义完整）  ←→  chunk 不能太大（检索精度）
                     ←→  chunk 不能太小（信息足够）
```

**策略：边界优先 + 阈值后悔**

```
对每段内容：
  第一步：按结构边界切（章节/段落/句子）
  第二步：如果这段内容 > max_chunk_size →
           在它内部递归，按下一级结构边界继续切
  第三步：如果这段内容 < min_chunk_size →
           和相邻的段落合并（如果合并后不超过 max_chunk_size）
```

| 参数 | 作用 | 典型值 |
|------|------|--------|
| `max_chunk_size` | 硬上限，超过必定拆分 | 800-1200 tokens |
| `min_chunk_size` | 低于此值尝试合并 | 200-300 tokens |

### 切多大？经验法则

| chunk_size | 适合的场景 | 问题 |
|-----------|-----------|------|
| 128-256 tokens | 问答对、代码片段、定义类 | 容易丢失上下文 |
| 512-1024 tokens | 通用文档、技术文章 | **最常见的范围** |
| 1024-2048 tokens | 长段落、复杂说明 | 信息密度下降 |
| > 2048 tokens | 整节文档 | 检索精度显著下降 |

### 重叠（Overlap）

**为什么要重叠？** 如果切在「第二步」和「设置数据库」之间，Chunk 1 有「第二步」没有「设置数据库」，Chunk 2 有「设置数据库」没有「第二步」。谁都回答不了问题。

**解决方案：** chunk 之间留 10%-20% 的重叠。

```
chunk 1 (0-500): 「...第二步，设置数据库连接...」
chunk 2 (450-950): 「...第二步，设置数据库连接...重启服务」
```

### 高级技巧：Small-to-Big

检索用小 chunk（精确），回复用大 chunk（完整上下文）。

```python
# 索引阶段
small_chunks = splitter_small.split(document)   # 256 tokens
big_chunks = splitter_big.split(document)       # 1024 tokens

# 建立映射
small_chunk.big_chunk_id = big_chunk.id

# 检索阶段
retrieved_small = vector_db.search(query, k=5)
retrieved_big = [get_big_chunk(s.big_chunk_id) for s in retrieved_small]
# 把 big chunks 喂给 LLM
```

**为什么有效：** 小 chunk 的向量更「纯净」——不会被无关内容稀释。但小 chunk 的信息可能不完整——所以用大 chunk 来补充上下文。

## 4. Embedding——把语义翻译成几何

### 最原始的办法：词袋（Bag of Words）

把句子里每个词出现的次数当向量：

```
「苹果很好吃」→ [1, 1, 1, 0, 0, 0]  （苹果=1, 很=1, 好吃=1, 甜=0, ...）
「苹果很甜」  → [1, 1, 0, 1, 0, 0]
「火箭发射了」→ [0, 0, 0, 0, 1, 1]
```

**问题：** 「好吃」和「甜」是两个完全独立的维度。哪怕意思几乎相同，向量距离也不算特别近。词袋法不知道词和词之间的关系。

### 核心洞察：一个词由它周围的词定义

> "You shall know a word by the company it keeps." — J.R. Firth, 1957

「苹果」经常和「好吃」「甜」「脆」「水果」一起出现。
「火箭」经常和「发射」「燃料」「轨道」一起出现。

一个词的意思，本质上是由它周围出现的词定义的。

### Embedding 是怎么学的？——一个猜词游戏

给你一个句子，中间挖了一个空：

> 「我今天买了一个____，又红又甜，很好吃。」

九成人会填「苹果」。没人会填「火箭」。

**embedding 模型就是在玩这个游戏——玩几十亿次。**

```
有词 A = 「苹果」
取包含「苹果」的真实句子 → 「我今天买了一个苹果，又红又甜，很好吃」

训练任务：
给模型看「我今天买了一个____，又红又甜，很好吃」
→ 模型需要填「苹果」
→ 填错了则调整参数，让正确路径更稳固
```

玩了几十亿次之后（整个语料库的每一句话都切成这样的猜词游戏），模型里每个词对应的向量会变成什么样？

- 「苹果」和「香蕉」会变得很近——它们经常出现在相似的上下文里
- 「苹果」和「火箭」会变得很远——它们周围出现的词几乎没有重叠

**你不需要告诉模型「苹果」和「香蕉」是相似的——模型通过猜词游戏自己学到了。**

### 从词向量到句子向量

一个句子有 N 个词，每个词有一个向量。怎么变成一个句子向量？

- **最笨的办法：** 求平均——但丢失了词序信息（「猫追狗」和「狗追猫」的平均向量一样）
- **更好的办法：** 用 Transformer——它让每个词通过注意力机制向周围的词「打招呼」，根据上下文调整自己的含义

> 「苹果很好吃」中的「苹果」→ 向量 A
> 「苹果公司发布了新手机」中的「苹果」→ 向量 B

A 和 B 不一样。因为 Transformer 知道前一个「苹果」身边是「好吃」（水果），后一个「苹果」身边是「公司」「发布」（企业名）。

这就是 **上下文感知的 embedding（contextual embedding）**。

## 5. Embedding 模型的结构

### 现代 embedding 模型到底长什么样？

以开源的 **E5-mistral-7b** 为例：

```
输入: 「我今天买了一个苹果」
    ↓
Tokenizer（把文字变成 token ID）
    ↓
[101] [451] [321] [789] ... [102]
    ↓
Token Embedding Layer（把每个 token ID 查表变成向量）
    ↓
[vec₁] [vec₂] [vec₃] ... [vec₇]   ← 每个 token 一个向量
    ↓
32 层 Transformer（每层 = Attention + FFN）
    ↓
[vec₁'] [vec₂'] [vec₃'] ... [vec₇']  ← 更新后的 token 向量
    ↓
Pooling（取最后一个 token / mean pooling）
    ↓
[一个向量, 4096 维]
    ↓
Projection Layer（Linear 层，4096 → 1536）
    ↓
L2 归一化（让向量长度 = 1）
    ↓
[最终 embedding, 1536 维]
```

### Decoder-only 能做 embedding 吗？

能。现代 SOTA embedding 模型大部分基于 **decoder-only** 架构（GPT 那一路）。

传统的直觉认为 encoder（BERT）更好因为它能看到完整的上下文。但实际中，decoder-only 模型在预训练阶段看了更多的数据（几万亿 token vs 千亿），数据量碾压了架构的微弱劣势。

### Embedding 模型 vs 普通 LLM

| 组件 | 普通 LLM | Embedding 模型 |
|------|----------|---------------|
| 最后输出 | 每个 token → 预测下一个词 | 一个向量 → 代表整个序列 |
| Pooling | 不需要 | 必须有（从 N → 1） |
| 投影层 | 无 | 有（隐藏维度 → embedding 维度） |
| 训练方式 | 预测下一个 token | 对比学习 |

## 6. Embedding 模型怎么训练

### 一个「鸡生蛋蛋生鸡」的问题

问：在没有 embedding 模型的时候，第一层的 Token Embedding 查找表从哪里来？

**答：随机数。**

词表大小 50,000，隐藏维度 4096，就分配一个 50,000 × 4096 的矩阵，每个数从正态分布中随机采样。第一次 forward 的时候，「苹果」对应的向量完全是随机的。

然后第一次训练——loss 告诉你「这个位置不对」，梯度反向传播，查找表的参数被更新了一点点。

经过几十万步训练，这些随机数就慢慢变成了有意义的向量。

**没有鸡生蛋蛋生鸡。只有一个从随机到语义的自举过程。**

### 完整链条

```
随机数
    ↓
LLM 预训练（猜下一个词，几万亿 token）
    ↓ 有了可用的 token 向量
对比学习微调（拉近正样本，推远负样本）
    ↓
Embedding 模型
```

### 训练数据长什么样

每一行是：(query, positive_doc, [negative_docs])

```
Q: 「怎么重置密码」  → 正例: 「密码重置步骤：进入设置...」
Q: 「苹果公司 CEO」 → 正例: 「蒂姆·库克于 2011 年担任...」
```

行业标准数据集：**MS MARCO**（来自必应搜索的真实数据）、**SNLI / MNLI**（自然语言推理数据）。

### 训练的核心损失函数：InfoNCE Loss

在一个 batch 里，有 N 个 (query, positive_doc) 对：

```
Q1 → D1+（正样本）
Q2 → D2+
...
QN → DN+

对 Q1 来说：
  正样本：D1
  负样本：这个 batch 里除了 D1 以外的所有其他文档
```

**InfoNCE Loss 做的事就是：让模型学会把 Q 和它的正样本 D+ 的距离拉近，同时把 Q 和其他所有负样本 D- 的距离推远。**

公式看起来复杂，但直觉很简单：

```
Loss = -log( Q和D+的相似度 / (Q和D+的相似度 + Q和所有D-的相似度之和) )
```

- 如果 Q 和 D+ 非常近，和其他 D 都非常远 → 比值 ≈ 1 → Loss ≈ 0（模型对了）
- 如果 Q 和 D+ 中等，和某个 D- 也挺近 → 比值 ≈ 0.4 → Loss ≈ 0.9（模型要调整）

### 训练时更新哪些参数？

| 策略 | 更新范围 | 效果 | 成本 |
|------|---------|------|------|
| **全参数微调** | 所有参数（7B） | 最好 | 最高 |
| **LoRA** | 只训练少量低秩矩阵 | 接近全参 | 低（单卡可跑） |
| **只训练投影层** | 只训练最后的线性投影层 | 有限 | 最低 |

**工业界实践中，大多数做 embedding fine-tuning 用的是 LoRA。** 你不需要改变模型对语言的基础理解，只需要调整它在「语义压缩」这个任务上的表现。

## 7. 向量数据库——存什么、怎么查

### 向量数据库存的不是「只有向量」

向量数据库里的一行数据：

```json
{
  "id": "doc1_chunk_5",
  "vector": [0.023, -0.415, 0.178, ..., 0.891],
  "payload": {
    "text": "配置数据库的步骤：第一步...",
    "source": "部署手册.pdf",
    "page": 42,
    "chapter": "3.2 数据库配置"
  }
}
```

**搜索时，返回的是整条记录，不是单独的向量。** 向量是用来比较的，payload 才是你要用的内容。

```
query 向量 → 在库里找最近的 5 个向量
    ↓
返回 5 条记录（每条包含 vector + payload）
    ↓
只取 payload.text，拼进 prompt 给 LLM
```

### 内部是怎么搜的？——HNSW 索引

如果你把几百万个向量存成一个巨大的列表，每次 query 来了挨个算距离——那就是暴力搜索。准确但慢。

**HNSW（Hierarchical Navigable Small World）** 用一种更聪明的方法：

```
  第 3 层（最稀疏）    ●━━━━●━━━━●━━━━●       ← 跳过大范围区域
  第 2 层            ●━━●━●━━●━●━━●━●      ← 缩小范围
  第 1 层（最密集）   ●●━●━●●━●━●●━●━●●     ← 精确搜索
```

**搜索过程：**

```
query 向量
    ↓
从第 3 层最高的入口点进入
    ↓
在当前层，贪心地走向最近邻的点
    ↓
走到局部最优（找不到更近的邻居了）
    ↓
降一层，继续走
    ↓
在第 1 层精确地找出最近的 k 个点
    ↓
取出这些点上的 payload（原始文本）
```

**时间复杂度大约 O(log N)，几百万个向量搜一次只要几毫秒。**

### 另一个主流索引：IVF（Inverted File Index）

```
所有向量
    ↓
K-means 聚类 → 分成 N 个簇
    ↓
每个簇有一个「质心」（centroid）
    ↓
搜索时：先算出 query 离哪个质心最近 → 只搜那个簇
```

**为什么快：** 100 万个向量分成 100 个簇，每个簇约 1 万个。你只需要算 1 次距离找到最近的簇，然后在 1 万而不是 100 万里搜。

## 8. Reranker——海选之后的面试

### 为什么有了向量检索还需要 reranker？

**根本原因：向量检索和 reranker 在「看」的方式上有本质不同。**

**向量检索（Dense Retrieval）：**

```
query → 编码成一个向量
chunk → 编码成一个向量
匹配分数 = 两个向量的点积
```

**每个 chunk 在编码时是独立的，不知道 query 长什么样。** 它在猜「什么样的 query 会找到我」——但猜得不一定准。

**Reranker（Cross-Encoder）：**

```
输入: 「怎么配置数据库连接？ [SEP] 打开配置文件，修改 host [SEP]」
    ↓
BERT 类模型 →
  「配置」(query 中的) 和「配置文件」(chunk 中的) 发生了注意力交互
  模型直接看到「哦，它们是在说同一件事」
    ↓
输出一个匹配度分数
```

**关键差异：reranker 里 query 和 chunk 的每个词都互相看到了。** 这种交互是向量检索完全做不到的。

### 为什么不用 reranker 做全部检索？

**原因只有一个：计算成本差了几个数量级。**

| | 向量检索 | Reranker |
|---|---|---|
| query 编码 | 1 次 | 每次都要拼接编码 |
| chunk 编码 | 1 次（提前编码好） | 每次都要重新编码 |
| 匹配计算 | 点积 ≈ O(d) | Transformer 推理 ≈ O(L² d) |
| 耗时（100 万 chunk） | ~50ms | 不可行 |
| 可缓存 | chunk 向量可提前算好 | 不能缓存 |

**两阶段架构产生的根本原因是一个经济决策：廉价的海选 + 昂贵的面试。**

```
向量检索 → 海选出 top-100 ~ top-200（快，覆盖面广）
    ↓
Reranker → 从候选里挑出 top-3 ~ top-10（慢，但精度高）
    ↓
给 LLM
```

### Reranker 怎么训练的？

训练数据：(query, chunk_positive, chunk_negative)

**核心思想：不是让模型输出一个「绝对分数」，而是让它在正样本和负样本之间拉开差距。**

```
loss = -log(sigmoid(score_positive - score_negative))
```

**关键技巧：Hard Negative Mining（硬负例挖掘）**

- **Easy negative**：query「怎么配置数据库」, 负例「四川火锅的做法」→ 太简单，学不到东西
- **Hard negative**：query「怎么配置数据库」, 负例「数据库的原理介绍」→ 看起相关但不是用户要的——这才是 reranker 需要学习的能力

向量检索的前 k 个 chunk 里，大部分「假阳性」恰恰是这种「看着相关其实不是」的——表面有同一个词但意图不匹配。

### Reranker 的典型效果

| 阶段 | NDCG@10 |
|------|----------|
| 仅向量检索（无 reranker） | 0.52 |
| + 简单 reranker | 0.58 |
| + 强 reranker | 0.62 |
| + 领域微调 reranker | 0.65-0.70 |

**关键的公式：**

```
最终检索质量提升 ≈ (reranker 质量) × (海选阶段的 recall)
```

reranker 再好，如果海选阶段就把正确答案筛掉了，也救不回来。

## 9. Hybrid Search——语义和关键词的联姻

### 向量检索的盲区

向量检索擅长「意思相近」，但不擅长「精确匹配」。

```
文档里有一句话： 「用户的初始密码重置链接将在 24 小时后过期」
用户问：         「密码重置链接过期时间」
```

向量检索可能能找到，也可能找不到——「过期时间」和「24 小时后过期」语义上不完全匹配。

**那「精确匹配」是什么技术做的？BM25（Best Matching 25）。**

### BM25 的直觉

BM25 做的事情本质上很简单：**在文档里找出现 query 里这些词最多的 chunk，同时考虑这个词在整个文档库里多常见。**

```
query: 「密码重置链接过期时间」
拆成词: [密码, 重置, 链接, 过期, 时间]

chunk_A: 「密码重置链接将在 24 小时后过期」→ 命中 4/5 词 → 高分
chunk_B: 「密码强度要求至少 8 位字符」    → 命中 1/5 词 → 低分
```

**BM25 靠关键词精确命中来做匹配。它没有语义理解，但它在精确匹配上做得非常好。**

### 为什么混合

| 检索方式 | 擅长 | 不擅长 |
|---------|------|--------|
| 向量检索（Dense） | 语义相似、同义词替换 | 稀有词精确匹配 |
| BM25（Sparse） | 关键词精确匹配 | 语义泛化 |

**它们在对方薄弱的地方强。所以两个都用，把结果合并。**

### RRF（Reciprocal Rank Fusion）——最常用的合并方法

**核心思想：把排名转成分数，而不是直接加原始分数。**

```
每个 chunk 的最终分数 = 1/(k + rank_dense) + 1/(k + rank_bm25)

其中 k 通常取 60
```

**为什么不用原始分数？** 向量检索的分数是 0-1 的余弦相似度，BM25 的分数是 0-几十的加权和。量级不同，不能直接加。但排名（rank）是统一的。

**举例：**

```
chunk_A: 向量检索排名第 3, BM25 排名第 50
  RRF: 1/(60+3) + 1/(60+50) = 0.0159 + 0.0091 = 0.0250

chunk_B: 向量检索排名第 40, BM25 排名第 2
  RRF: 1/(60+40) + 1/(60+2) = 0.0100 + 0.0161 = 0.0261  ← 胜出
```

chunk_B 虽然向量检索排名只有 40，但因为 BM25 排名非常高（2），最终分数反而更高。

### 一句话总结

> 向量检索问「意思一样吗？」，BM25 问「词一样吗？」。Hybrid Search 是「两个都问，结果合并」。

## 10. MMR 去重——别让 LLM 看重复的内容

### 又一个问题

你用向量检索 + reranker，拿到了 5 个最相关的 chunk：

```
Chunk 1: 「配置数据库的步骤：第一步，打开配置文件...」
Chunk 2: 「打开配置文件后，需要修改数据库连接字符串...」
Chunk 3: 「数据库连接字符串的格式为 jdbc:mysql://...」
Chunk 4: 「配置文件中找到 database.host 字段...」
Chunk 5: 「配置完成后，重启服务使新配置生效...」
```

chunk 1、2、3、4 全都在说「配置数据库」这件事。信息高度重叠。LLM 的上下文窗口里，大部分是冗余信息。

**MMR（Maximum Marginal Relevance）要解决的问题：如何在保证相关性的同时，保证 chunk 之间的多样性。**

### MMR 的直觉

> 每选一个 chunk，既要考虑它跟 query 的相关性，又要考虑它跟已经选中的 chunk 不重复。

**边际收益 = 这个 chunk 带来的「新信息」减去「重复信息」。**

```
C2 的边际收益 = λ × sim(C2, query) - (1-λ) × sim(C2, C1)

λ=1   → 只看相关性（就是普通 top-k）
λ=0   → 只看多样性（不考虑 query）
λ=0.5 → 平衡
```

**举例（λ=0.7）：**

```
C2: 0.70 × 0.92 - 0.30 × 0.85 = 0.389
C3: 0.70 × 0.88 - 0.30 × 0.20 = 0.556  ← C3 跟 C1 不重复，胜出！
C4: 0.70 × 0.85 - 0.30 × 0.75 = 0.370
C5: 0.70 × 0.80 - 0.30 × 0.10 = 0.530

选 C3（虽然相关性不如 C2，但因为跟 C1 不重复，边际收益更高）
```

### λ 参数怎么选？

| 场景 | λ 推荐值 |
|------|---------|
| 事实性问答（「X 的值是什么？」） | 0.7-0.9 |
| 综述性问题（「介绍一下 X」） | 0.5-0.7 |
| 对比类问题（「X 和 Y 的差异」） | 0.3-0.5 |

### MMR vs Reranker 的关系

| | Reranker | MMR |
|--|---------|-----|
| **目标** | 找到跟 query 最相关的 chunk | 找到信息不重复的 chunk |
| **看什么** | 每个 chunk 独立跟 query 匹配 | chunk 之间的相互关系 |
| **如果只有它** | 选出的 chunk 可能全是冗余信息 | 可能选到不相关的 chunk |

**实际分工：**

```
向量检索 → top-200    （海选，只求覆盖率）
    ↓
Reranker → top-20     （淘汰不相关的，排好序）
    ↓
MMR      → top-5      （在 top-20 里选信息最多样化的 5 个）
    ↓
LLM
```

## 11. Agentic RAG vs 普通 RAG——怎么选

### 两种 RAG 在做什么

**普通 RAG（老实人 RAG）：**

```
用户问 → 查一次 → 拼 prompt → LLM 回答
```

三步。干净、可预测。

**Agentic RAG（侦探模式）：**

```
用户问 → LLM 看情况决定：
          要不要先改写查询？
          要不要查多次？
          要不要先问用户澄清？
         → 回答或追问
```

### 三个决策维度

#### 维度一：静态射击 vs 动态追击

**静态射击：** 问题独立、具体、直接对应文档中的某个段落。

- 「2024 年苹果的营收是多少？」
- 「这个错误码是什么意思？」

**→ 普通 RAG 就够了。**

**动态追击：** 需要多步推理、信息聚合、在多个不明确的可能性中找答案。

- 「对比苹果和谷歌过去五年 AI 收购策略的差异」
- 「我想做聊天机器人，但不确定需要什么架构」

**→ 考虑 Agentic RAG。**

#### 维度二：文档库结构

**普通 RAG 喜欢的：** 结构一致、信息相对独立、每个 chunk 包含完整信息。

**Agentic RAG 的必要场景：** 信息分散、多种格式（文本/表格/代码）、不同文档有矛盾。

#### 维度三：可预测性要求

**普通 RAG：** 行为可预测，同样的查询输出同样结果，测试和调试直接。

**Agentic RAG：** 行为发散，LLM 的「思考」过程每次不同，可能反问用户。

### 决策表

| 场景 | 推荐 | 理由 |
|------|------|------|
| FAQ 式问答 | **普通 RAG** | 多一步都是浪费 |
| 需要从 3 个以上来源拼凑信息 | **Agentic RAG** | 普通 RAG 会漏 |
| 用户提问长、具体 | **普通 RAG** | 一次检索就够了 |
| 用户提问模糊、简短（「说说这个」） | **Agentic RAG** | 需要主动理解意图 |
| 延迟敏感（<1 秒） | **普通 RAG** | Agentic 推理循环太慢 |
| 可接受 3-5 秒 | **Agentic RAG** | 值得换质量 |
| 团队小、维护能力有限 | **普通 RAG** | Agentic 调试成本高 |

### 实践建议：从普通 RAG 开始

大多数团队都会过度设计。更好的路径是：

1. **先用普通 RAG**，认真做评估
2. 抓出**失败模式**——哪些问题答不好？
3. 如果失败原因是「信息够了但推理不够」→ 优化 prompt
4. 如果失败原因是「检索到的信息不完整」→ 优化检索（reranker、hybrid search）
5. 如果失败原因是「需要从多处分步获取信息，且当前步骤依赖上一步的结果」→ **这才是你真的需要 Agentic RAG 的信号**

**不要一开始就上 Agentic RAG。那应该是你优化出来的结果，不是起点。**

## 12. 评估 RAG 系统

### 检索层面——你找对东西了吗？

- **Hit Rate / Recall@k**：正确答案在不在你检索回来的 top-K 里？
- **MRR（Mean Reciprocal Rank）**：正确答案排在第几位？
- **NDCG（Normalized Discounted Cumulative Gain）**：考虑排序完整性和相关性等级

### 生成层面——你回答对了吗？

- **Faithfulness（忠实度）**：回答有没有编造信息？有没有说参考资料里没有的东西？
- **Answer Relevancy（回答相关性）**：回答有没有回答到用户的提问？
- **Context Precision**：你给的资料有多少是真正有用的？

**常用评估框架：RAGAS、TruLens、LangSmith。**

## 13. 完整检索链路一览

```
用户 query
    ↓
Query Rewriting（改写/扩展模糊查询）
    ↓
并行检索：
  ├── Dense Retriever（向量检索）→ top-200
  └── Sparse Retriever（BM25）   → top-200
    ↓
RRF 合并 → top-200
    ↓
Reranker → top-20（重排序，淘汰假阳性）
    ↓
MMR → top-5（去重，保证信息多样性）
    ↓
提供给 LLM 作为上下文
    ↓
LLM 生成回答
```

**每一步解决不同的问题：**

| 步骤 | 解决的问题 | 输入 | 输出 |
|------|-----------|------|------|
| Dense + Sparse | 语义泛化不够？精确匹配不够？ | 全部文档 | 各 200 个候选 |
| RRF | 两个系统的分数尺度不统一 | 两个 200 名榜单 | 合并后的 200 名 |
| Reranker | 点积匹配太肤浅 | 200 个候选 | 20 个高质量 chunk |
| MMR | 信息重复浪费上下文 | 20 个高质量 chunk | 5 个多样化的 chunk |

## 写在最后

RAG 的技术细节很多，但核心逻辑其实不复杂：

1. **把文档切成能查的块**（分块策略决定了系统的上限）
2. **把块变成能比的向量**（embedding 质量决定了检索的精度）
3. **快速找到相关的块**（向量索引决定了系统的速度）
4. **从候选里精挑细选**（reranker + hybrid + MMR 决定了下游的质量）
5. **把资料给 LLM，让它回答**（prompt 设计决定了最终表达）

每一步都有工程技巧和权衡，没有万能方案。最好的做法是：**从最简单的开始，跑起来，观察失败模式，然后针对性地优化。**
